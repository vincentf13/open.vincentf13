# 現貨交易系統測試計畫 (Spot Exchange Test Plan)

## 1. 壓力測試計畫 (Stress Test Plan)

### 目的
測出真實場景下的最大每秒交易量 (Maximum TPS)。

### 測試內容
- **吞吐量測試**：模擬真實用戶行為（下單、撤單、查詢），逐步增加併發請求，找出系統處理能力的極限。
- **反應時間監控**：在高負載情況下，紀錄核心 API（如撮合撮合、訂單簿更新）的 99th 百分位延遲。
- **資源佔用分析**：監控 CPU、記憶體、硬碟 I/O 以及網路頻寬在壓力峰值時的表現，識別潛在瓶頸。

---

## 2. 可靠性測試計畫 (Reliability Test Plan)

### 目的
測出各種極端情況下，系統容災恢復後數據的完備性、一致性與可靠性。

### 測試內容
- **數據完備性**：在系統崩潰或意外停機重啟後，確認所有已撮合的成交記錄、資產餘額均無遺漏。
- **數據一致性**：驗證訂單中心、撮合引擎與清算系統之間的數據是否完全匹配，無溢出或偏差。
- **故障恢復 (Failover)**：模擬服務節點當機、資料庫斷連、網路分區等場景，測試系統自動切換與自我修復的能力，確保最終一致性。

---

## 3. 具體最小可行測試方法 (MVP Test Methods)

### A. 壓力測試：k6 模擬路徑
1. **腳本邏輯**：
   - 模擬虛擬用戶 (VUs) 持續調用 `POST /api/v1/orders` (下單) 與 `GET /api/v1/orderbook` (查詢)。
   - 階梯式壓力：100 -> 500 -> 1000 -> 2000 VUs，每階持續 2 分鐘。
2. **監控與判定**：
   - 使用 Prometheus/Grafana 監控 Pod 資源佔用。
   - 當 **HTTP 錯誤率 > 1%** 或 **P99 延遲 > 500ms** 時，前一階段的平均 TPS 即為系統瓶頸。

### B. 可靠性測試：Pod 注入故障 (Fault Injection)
1. **故障場景**：
   - 在高負載（500 TPS）運行中，執行 `kubectl delete pod -l app=matching-engine` 強制重啟撮合引擎。
   - 模擬資料庫瞬斷（重啟 MySQL Pod）。
2. **數據校驗 (SQL 檢查)**：
   - **資產平帳**：`SELECT SUM(balance) FROM user_assets` 需與故障前數據扣除成交額後一致。
   - **狀態一致性**：比對撮合引擎 Redis 中的最後 Sequence ID 與資料庫中 `trades` 表的最後一筆 ID，確認無重疊或遺失。
   - **訂單流水**：確認所有已扣款但未成交的訂單在系統恢復後，狀態正確（未被誤設為已成交或遺失）。
